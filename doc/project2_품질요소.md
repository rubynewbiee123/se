# 프로젝트 II - 대상 시스템 품질 요소 추정

## NextDriv 시스템 품질 요소 분석

1장의 프로젝트 정의서에서 제시한 핵심 기능과 유사 시스템 분석 결과를 기반으로, 대상 시스템에서 중요하게 고려해야 할 품질 요소를 추정하고 정리합니다.

---

## (1) 사용자가 원하는 기대치

### 🎯 핵심 가치: 편리성, 안정성, 보안성

#### 1. 직관적인 사용성
별도 교육 없이 파일 업로드/다운로드, 폴더 관리 등 기본 기능을 즉시 수행할 수 있어야 한다.

| 측정 항목 | 목표 기준 |
|-----------|-----------|
| 신규 사용자 주요 기능 수행 소요 시간 | 최초 접속 후 5분 이내 완료 가능 |
| 주요 작업 클릭 수 | 파일 업로드 3클릭 이내, 공유 설정 4클릭 이내 |

#### 2. 빠른 응답 속도
파일 검색, 페이지 로딩, 대용량 파일 전송 등 모든 작업에서 체감 지연이 없어야 한다.

| 측정 항목 | 목표 기준 |
|-----------|-----------|
| 페이지 초기 로딩 시간 | 3초 이내 |
| 파일 검색 응답 시간 | 1초 이내 (1,000개 파일 기준) |
| 동시 접속 200명 시 응답 지연 | 평균 대비 20% 이내 증가 |

#### 3. 안정적인 파일 관리
데이터 손실 없이 파일을 안전하게 보관하고, 오류 발생 시 자동 복구되어야 한다.

| 측정 항목 | 목표 기준 |
|-----------|-----------|
| 파일 업로드 성공률 | 99.9% 이상 |
| 버전 이력 보관 | 최소 30일, 최대 10개 버전 유지 |
| 업로드 실패 시 자동 재시도 | 최대 3회 |

#### 4. 안전한 공유
사용자별 권한 설정, 공유 링크 만료/비밀번호 보호, 접근 이력 확인이 가능해야 한다.

| 측정 항목 | 목표 기준 |
|-----------|-----------|
| 권한 변경 반영 시간 | 5초 이내 |
| 접근 이력 조회 가능 기간 | 최소 90일 |

#### 5. 동시 접근 충돌 처리
여러 사용자가 동일 파일에 동시에 접근할 때 데이터 충돌 없이 안전하게 처리되어야 한다.

| 측정 항목 | 목표 기준 |
|-----------|-----------|
| 파일 잠금(Lock) 반영 시간 | 다른 사용자가 수정 중일 때 즉시(3초 이내) 잠금 상태 표시 |
| 충돌 알림 | 동시 수정 시도 시 실시간 알림 제공 |
| 충돌 발생 시 데이터 손실 | 0건 (양쪽 버전 모두 보존) |
| 잠금 해제 사전 경고 | 비활성 상태 8분 경과 시 "2분 후 잠금이 해제됩니다" 경고 알림 표시 |
| 잠금 자동 해제 | 비활성 상태 10분 경과 시 자동 잠금 해제, 해제 시점에 현재 작업 내용을 임시 버전으로 자동 저장 |
| 잠금 해제 후 저장 시도 | 잠금 해제 이후 다른 사용자가 수정한 이력이 있을 경우, 덮어쓰기 없이 충돌 감지 후 버전 비교 화면 제공 |

#### 6. 데이터 용량 및 저장 공간
사용자별 충분한 저장 공간이 보장되며, 파일 크기 제한이 업무에 지장을 주지 않아야 한다.

| 측정 항목 | 목표 기준 |
|-----------|-----------|
| 사용자당 기본 할당 용량 | 10GB 이상 |
| 단일 파일 최대 업로드 크기 | 2GB |
| 저장 용량 사용률 알림 | 80%, 90%, 100% 도달 시 사용자에게 알림 |

## (2) 개발자 혹은 유지보수 담당자가 원하는 기대치

### 🔧 핵심 가치: 유지보수성, 확장성, 안정성

#### 1. 유지보수 용이성
모듈화된 코드 구조와 명확한 컨벤션으로 신규 인력 투입 시 빠르게 적응할 수 있어야 한다.

| 측정 항목 | 목표 기준 |
|-----------|-----------|
| 신규 개발자 온보딩 기간 | 1주일 이내 기본 수정 가능 |
| 코딩 컨벤션 위반 | ESLint/Prettier 기준 0건 |
| 모듈 간 순환 의존성 | 0건 |

#### 2. 확장 가능성
신규 기능 추가 시 기존 코드 수정을 최소화할 수 있는 구조여야 한다.

| 측정 항목 | 목표 기준 |
|-----------|-----------|
| 신규 API 엔드포인트 추가 소요 시간 | 1일 이내 (단순 CRUD 기준) |
| 신규 기능 추가 시 기존 모듈 변경 | 2개 이내 |

#### 3. 안정성 및 테스트
예외 처리, 에러 로깅, 자동화 테스트를 통해 장애를 최소화하고 빠르게 대응할 수 있어야 한다.

| 측정 항목 | 목표 기준 |
|-----------|-----------|
| 평균 장애 복구 시간 (MTTR) | 1시간 이내 |
| 평균 장애 간격 (MTBF) | 720시간(30일) 이상 |
| 단위 테스트 코드 커버리지 | 70% 이상 |
| CI/CD 빌드 및 배포 소요 시간 | 10분 이내 |

#### 4. 성능 최적화
DB 쿼리 최적화, 캐싱, 스트리밍 처리를 통해 200명 동시 사용 환경에서 안정적으로 동작해야 한다.

| 측정 항목 | 목표 기준 |
|-----------|-----------|
| 동시 접속자 수 | 200명 정상 처리 (피크 시 300명) |
| API 평균 응답 시간 | 200ms 이내 |
| 서버 CPU 사용률 | 평상시 40% 이내, 피크 시 70% 이내 |

#### 5. 동시 접근 제어 구현
파일 잠금(Lock) 메커니즘과 충돌 방지 로직을 안정적으로 구현하고, 예외 상황에 대한 처리가 명확해야 한다.

| 측정 항목 | 목표 기준 |
|-----------|-----------|
| 파일 잠금 API 응답 시간 | 100ms 이내 |
| 데드락(Deadlock) 발생 | 0건 |
| 잠금 상태 정합성 | 비정상 종료 시에도 잠금 자동 해제 보장 |

#### 6. 스토리지 구현 효율성
파일 저장/삭제 처리의 기술적 효율성을 확보하고, 스토리지 백엔드 변경 시에도 코드 수정 없이 대응할 수 있는 구조여야 한다.

| 측정 항목 | 목표 기준 |
|-----------|-----------|
| 중복 파일 탐지 | 해시(SHA-256) 기반 중복 파일 감지로 저장 공간 절약 |
| 삭제 파일 처리 | 휴지통 보관 30일 후 자동 영구 삭제 (스케줄러 구현) |
| 스토리지 백엔드 교체 시 코드 수정 | 설정 파일 변경만으로 대응 (코드 수정 0건) |
| 대용량 파일 업로드 처리 | 청크(Chunk) 단위 분할 업로드 지원 |

## (3) 개발 조직의 관리자가 원하는 기대치

### 📊 핵심 가치: 비용 효율성, 일정 준수, 보안 규정 준수

#### 1. 비용 효율성
오픈소스 기반 자체 구축으로 상용 클라우드 서비스 대비 운영 비용을 절감해야 한다.

| 측정 항목 | 목표 기준 |
|-----------|-----------|
| 라이선스 비용 | 0원 (전체 오픈소스 스택) |
| 월 운영 비용 | 동일 규모 SaaS 대비 50% 이하 |

#### 2. 보안 및 규정 준수
데이터 암호화, 인증/권한 관리, 관련 법규 준수를 통해 조직 보안 정책을 100% 충족해야 한다.

| 측정 항목 | 목표 기준 |
|-----------|-----------|
| 데이터 전송 암호화 | TLS 1.2 이상, 100% HTTPS |
| 로그인 실패 제한 | 5회 실패 시 15분 계정 잠금 |
| 보안 취약점 점검 주기 | 분기별 1회 이상 |

#### 3. 확장성 및 운영 안정성
현재 200명에서 500명 이상으로 무중단 확장이 가능하고, 실시간 모니터링과 백업 체계를 갖춰야 한다.

| 측정 항목 | 목표 기준 |
|-----------|-----------|
| 최대 지원 사용자 수 | 500명 이상 (아키텍처 변경 없이) |
| 장애 알림 발송 시간 | 감지 후 1분 이내 |
| 목표 복구 시간 (RTO) | 4시간 이내 |
| 목표 복구 시점 (RPO) | 24시간 이내 |
| 백업 주기 | 일일 증분, 주간 전체 백업 |

#### 4. 스토리지 용량 계획 및 비용 관리
전체 시스템의 스토리지 용량을 조직 차원에서 체계적으로 계획하고, 증설 시기와 예산을 사전에 예측할 수 있어야 한다.

| 측정 항목 | 목표 기준 |
|-----------|-----------|
| 전체 시스템 초기 스토리지 | 최소 2TB (200명 × 10GB 기준) |
| 스토리지 증설 기준 | 전체 용량 70% 도달 시 증설 검토 |
| 용량 증설 소요 시간 | 무중단 증설, 4시간 이내 완료 |
| 사용량 추이 리포트 | 월 1회 자동 생성 |
| 연간 스토리지 비용 예측 | 사용량 추이 기반 다음 분기 비용 예측 리포트 제공 |

## 품질 요소 우선순위

### 최우선
| 순위 | 품질 요소 | 분류 근거 |
|------|-----------|-----------|
| 1 | **보안성** | 조직 내부 파일을 다루는 시스템 특성상 데이터 유출 시 비즈니스에 치명적 손해가 발생하며, 보안 사고는 사후 복구가 사실상 불가능하므로 최우선으로 보장해야 한다. |
| 2 | **안정성** | 파일 스토리지의 핵심 목적이 데이터의 안전한 보관이므로, 데이터 손실이나 시스템 장애는 서비스 신뢰도를 근본적으로 훼손한다. |
| 3 | **성능** | 200명이 동시에 사용하는 환경에서 응답 지연은 곧 업무 생산성 저하로 직결되며, 성능 문제는 사용자 이탈의 가장 주된 원인이 된다. |

### 높음
| 순위 | 품질 요소 | 분류 근거 |
|------|-----------|-----------|
| 4 | **사용성** | 시스템이 안정적으로 동작하더라도 사용이 어려우면 실질적 활용률이 떨어지고, 기존 상용 서비스(Google Drive, Dropbox) 대비 경쟁력을 확보하기 어렵다. |
| 5 | **유지보수성** | 초기 개발 이후 장기적인 운영 과정에서 코드 품질과 문서화 수준이 기능 개선 속도와 버그 대응 시간을 결정하는 핵심 요인이 된다. |

### 중간
| 순위 | 품질 요소 | 분류 근거 |
|------|-----------|-----------|
| 6 | **확장성** | 현재 200명 규모에서 당장 필요하지는 않으나, 조직 성장에 따라 500명 이상으로 확장해야 할 경우를 대비하여 아키텍처 수준에서 미리 고려해두어야 한다. |
| 7 | **비용 효율성** | 오픈소스 기반 자체 구축이라는 프로젝트 방향이 이미 비용 절감을 전제하고 있으므로, 추가적인 비용 최적화는 서비스 안정화 이후 단계에서 점진적으로 개선한다. |
