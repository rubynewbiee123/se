## 1. 개요(Introduction)

### 1.1 문서의 목적
이 문서는 조직 내 팀원들이 파일을 중앙에서 관리하고 공유할 수 있는 클라우드 기반 파일 관리 시스템 **NextDriv** 의 **요구사항 정의서(Requirements Definition Document, RDD)** 이다. 본 문서는 시스템이 제공해야 하는 기능적 요구사항, 비기능적 요구사항, 인터페이스 요구사항을 식별하고 정의하여, 이후 설계·구현·테스트 단계의 기준으로 활용하는 것을 목적으로 한다.

본 문서는 다음과 같은 범위를 다룬다.
- NextDriv 시스템이 수행해야 하는 기능 명세
- 200명 동시 접속 환경에서의 성능·보안·안정성 등 비기능 명세
- 프론트엔드와 백엔드, 외부 사용자와의 인터페이스 명세
- 프로젝트 정의서 및 품질 요소 분석에서 도출한 7가지 품질 요소 기반의 품질 요구사항

### 1.2 프로젝트 개요

#### 1.2.1 프로젝트 정의
NextDriv는 약 200명 규모의 조직 내 팀원이 동시 사용 가능한 웹 기반 클라우드 파일 관리 시스템이다. 사용자는 웹 브라우저를 통해 다양한 형식의 파일을 업로드·다운로드하고 폴더로 체계적으로 정리할 수 있으며, 팀원 간 파일 공유 및 권한 관리, 외부 협력 업체와의 보안 링크 공유, 파일 버전 관리 기능을 제공한다.

#### 1.2.2 주요 기능 설명

| ID | 설명 |
| :--- | :--- |
| F01 | 다양한 형식의 파일을 웹 브라우저에서 업로드·다운로드 |
| F02 | 폴더 생성·이름 변경·삭제 및 다단계 폴더 구조 지원 |
| F03 | 사용자별 열람/수정/댓글 권한 설정 및 외부 공유 링크 제공 |
| F04 | 파일명, 날짜, 업로더, 파일 유형 기준의 빠른 검색 |
| F05 | 수정 이력 보관, 이전 버전 조회 및 복원 |
| F06 | 관리자의 계정 생성/삭제, 저장 공간 관리 및 그룹 기능 |
| F07 | 다중 사용자 동시 접근 시 파일 잠금 및 충돌 처리 |
| F08 | 휴지통 기반 삭제 파일 보관 및 복구 |

### 1.3 용어 정의

| 용어 | 설명 |
| :--- | :--- |
| Chunk Upload | 대용량 파일을 작은 단위로 쪼개어 전송하여 네트워크 불안정 시 재전송 효율을 높이는 방식 |
| File Lock | 한 사용자가 파일을 수정하는 동안 다른 사용자의 동시 수정을 차단하는 동시성 제어 메커니즘 |
| RTO | Recovery Time Objective의 약자로, 장애 발생 후 시스템 복구까지의 목표 시간 |
| RPO | Recovery Point Objective의 약자로, 장애 시 허용 가능한 최대 데이터 손실 시점 |
| MTTR | Mean Time To Repair의 약자로, 평균 장애 복구 시간 |
| MTBF | Mean Time Between Failures의 약자로, 장애 사이의 평균 시간 |
| SHA-256 | 256비트 해시값을 생성하는 암호학적 해시 함수로, 본 시스템에서는 중복 파일 탐지에 사용 |
| TLS | Transport Layer Security의 약자로, 네트워크 통신 구간을 암호화하는 보안 프로토콜 |

본 장의 품질 요구사항은 프로젝트 II(NextDriv 시스템 품질 요소 추정)에서 식별한 7가지 품질 요소를 기준으로 정의한다. 우선순위는 동일 분석서의 분류를 따른다.

### 3.1 보안성(Security) — 최우선
조직의 중요한 데이터를 보호하고 외부 유출을 방지하기 위한 품질 요구사항이다.

| ID | 요구사항 | 우선순위 |
| :--- | :--- | :--- |
| QR-S-001 | 시스템은 모든 통신 구간에 HTTPS(TLS 1.2 이상)를 적용한다. | 상 |
| QR-S-002 | 시스템은 5회 연속 로그인 실패 시 계정을 15분간 잠근다. | 상 |
| QR-S-003 | 시스템은 분기별 1회 이상 보안 취약점 점검을 수행한다. | 상 |
| QR-S-004 | 시스템은 모든 인증 시도에 대해 감사 로그를 남긴다. | 상 |

### 3.2 안정성(Reliability) — 최우선
파일 스토리지의 핵심 목적인 데이터의 안전한 보관을 보장하기 위한 품질 요구사항이다.

| ID | 요구사항 | 우선순위 |
| :--- | :--- | :--- |
| QR-R-001 | 시스템의 파일 업로드 성공률은 99.9% 이상이다. | 상 |
| QR-R-002 | 시스템의 평균 장애 복구 시간(MTTR)은 1시간 이내이다. | 상 |
| QR-R-003 | 시스템은 일일 증분 백업 및 주간 전체 백업을 수행한다. | 상 |
| QR-R-004 | 시스템의 목표 복구 시간(RTO)은 4시간 이내이다. | 상 |
| QR-R-005 | 시스템의 목표 복구 시점(RPO)은 24시간 이내이다. | 상 |
| QR-R-006 | 시스템은 동시 수정 충돌 발생 시 데이터를 손실 없이 양쪽 버전을 모두 보존한다. | 상 |

200명 동시 사용 환경에서의 업무 생산성을 보장하기 위한 품질 요구사항이다.

| ID | 요구사항 | 우선순위 |

| QR-P-003 | 시스템의 파일 검색 응답 시간은 1초 이내이다. | 상 |
| QR-P-004 | 시스템의 서버 CPU 사용률은 평상시 40% 이내, 피크 시 70% 이내로 유지된다. | 중 |

## 5. 기타 고려 사항

- **법적/조직적 제약**: 사내 정보보안 정책에 따라 모든 사용자 활동 로그는 6개월 이상 보관해야 한다.
- **확장 로드맵**: 향후 다음 기능의 추가를 고려한다.
  - 문서 공동 편집 기능 (여러 사용자가 동시에 같은 문서를 편집)
  - 자동 파일 분류 기능 (파일 유형 및 내용에 따른 자동 분류)
  - AI 기반 검색 기능 (파일 내용 분석을 통한 정확한 검색 결과 제공)
- **TBD 항목**: SSO(Single Sign-On) 도입 여부 및 외부 인증(OAuth) 연동은 추후 결정한다.
- **변경 관리**: 요구분석 단계 종료 후 추가되는 기능은 원칙적으로 수용하지 않으며, 치명적 결함 발견 시 팀원 전원 동의 하에 범위를 조정한다.
